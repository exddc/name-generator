# TW-228 model cutover report — 2026-07-15

## Decision

Make `openai/gpt-oss-20b` the repository and deployment default. Use low
reasoning, strict JSON Schema output, no reasoning output, and no provider
streaming. Keep `openai/gpt-oss-120b` as an explicit whole-service emergency
rollback profile; routing only Creative requests to 120B is deliberately left
for a follow-up issue.

Groq lists Qwen 3 32B as a preview model retiring on 2026-07-17. GPT-OSS 20B
supports strict structured output and `low`, `medium`, or `high` reasoning.
Sources: [Groq deprecations](https://console.groq.com/docs/deprecations),
[GPT-OSS 20B model card](https://console.groq.com/docs/model/openai/gpt-oss-20b),
[structured outputs](https://console.groq.com/docs/structured-outputs), and
[API reference](https://console.groq.com/docs/api-reference).

## Versioned baseline and candidate smoke benchmark

- Repository baseline: `fb50dc4`
- Prompt set: `tw-228-v1` in `apps/api/scripts/benchmark_model_cutover.py`
- Workload: five representative LEGACY prompts, requesting 10 usable results
  while matching the application's 10-candidate over-generation behavior
- Runtime: Groq Python SDK 0.32.0; live GroqCloud calls from Berlin
- Prices captured on 2026-07-15: Qwen $0.29/M input and $0.59/M output;
  GPT-OSS 20B $0.075/M input and $0.30/M output

| Metric | Qwen 3 32B baseline | GPT-OSS 20B candidate | Delta |
| --- | ---: | ---: | ---: |
| Schema-valid responses | 5/5 | 5/5 | equal |
| Raw candidates | 100 | 100 | equal |
| Domain-syntax-valid candidates | 100% | 92% | -8 pp |
| Unique candidates | 100% | 98% | -2 pp |
| Mean latency | 12,504 ms | 5,303 ms | 57.6% lower |
| Median latency | 701 ms | 528 ms | 24.7% lower |
| Observed max / five-sample p95 | 41,668 ms | 24,455 ms | 41.3% lower |
| Input / output tokens | 1,556 / 837 | 2,094 / 835 | +538 / -2 |
| Estimated cost for five requests | $0.000945 | $0.000408 | 56.8% lower |

Both samples contained a provider queue outlier, so the five-sample p95 is a
smoke signal rather than a capacity estimate. The application now rejects
non-domain candidate strings before they reach the worker; GPT-OSS still
returned 92 usable candidates for 50 requested results. A sustained rejection
rate above 10% stops the canary.

## Contract and rollback evidence

The live production-like contract test retrieved the exact model ID and then
returned schema-valid candidates from `openai/gpt-oss-20b`:

```text
RUN_GROQ_CONTRACT_TEST=1 poetry run pytest tests/test_groq_contract.py -q -s
1 passed
```

Rollback profiles were also exercised through the application client, not a
raw curl request:

```text
qwen3-32b-rollback / qwen/qwen3-32b: 13 candidates, healthy
gpt-oss-120b-emergency / openai/gpt-oss-120b: 13 candidates, healthy
```

Qwen rollback is only valid before the 2026-07-17 shutdown. The 120B emergency
profile is the production-safe rollback after that point.

## Effective-container and end-to-end exercise

The full Compose stack built and started locally with Postgres, Redis, the
domain-check worker, API, and production Next.js web image. The API's startup
retrieval succeeded with the explicit Compose environment, and its metrics
reported the complete effective profile.

An authenticated request to the same `/v1/domain/stream` endpoint used by the
frontend completed the full chain: GPT-OSS 20B generated 12 accepted candidates
in 617 ms (419 input, 125 completion tokens), the worker checked them, and the
SSE response completed with two available domains. No candidates were rejected
and neither the prompt nor credentials appeared in logs.

The deployment configuration was then exercised in sequence:

1. `gpt-oss-20b` with `stage="canary"` became ready and appeared in
   `domain_generator_llm_model_info`.
2. The stop/rollback path recreated the API with
   `gpt-oss-120b-emergency`, `openai/gpt-oss-120b`, and `stage="rollback"`;
   startup passed and metrics showed the 120B effective model.
3. The API was restored to `gpt-oss-20b`, `stage="production"`, and all five
   local services remained healthy.

The nine Prometheus alert rules passed `promtool check rules`.
